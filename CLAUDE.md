# CLAUDE.md

## Project identity
이 프로젝트는 개인용 AI 비서 앱이 아니라, 로컬 PC를 실행 허브로 사용하는 자동화 플랫폼입니다.
사용자는 모바일/PC 프론트에서 자연어 또는 간단한 명령으로 요청하고,
시스템은 가능한 한 AI 사용을 최소화하면서도 AI 비서처럼 자연스럽게 동작해야 합니다.

## Core product philosophy
이 서비스의 핵심 차별점은 다음과 같습니다.

- AI를 과도하게 사용하지 않습니다.
- AI 사용량을 줄여 비용을 아끼는 것은 단순한 최적화가 아니라, 이 제품의 핵심 차별점입니다.
- 사용자는 AI가 모든 것을 처리하는 것처럼 느끼되,
  내부적으로는 가능한 한 규칙 기반 자동화와 오케스트레이션으로 처리합니다.
- AI는 꼭 필요한 순간에만 사용합니다.
- 목표는 "최소한의 AI로 최대한의 자동화 경험"입니다.
- 즉, 좋은 설계란 AI를 더 많이 붙이는 구조가 아니라, AI 없이 처리 가능한 범위를 최대한 넓히는 구조입니다.

## AI usage policy
AI는 아래 상황에서만 우선적으로 사용합니다.

- 사용자 의도 해석이 필요한 경우
- 요청이 모호하여 작업 계획 수립이 필요한 경우
- 여러 툴 또는 여러 단계 작업 중 적절한 흐름을 선택해야 하는 경우
- 결과를 자연어로 요약하거나 설명해야 하는 경우
- 예외 상황에서 대체 경로를 제안해야 하는 경우

AI를 사용하지 않고 처리해야 하는 대상은 다음과 같습니다.

- 정해진 스케줄 실행
- 크롤러 실행
- 데이터 수집 및 저장
- 캘린더 조회
- 메시지 발송
- 정책 검사
- 로그 기록
- 재시도 처리
- 파일 이동/복사
- 사전에 허용된 설치 스크립트 실행

항상 먼저 "이 작업을 AI 없이 처리할 수 있는가?"를 검토한 뒤,
정말 필요한 경우에만 AI를 사용합니다.

추가로 항상 다음 기준을 유지합니다.

- AI 호출 감소는 비용 절감뿐 아니라 제품 정체성 유지에 직접 연결됩니다.
- 새 기능을 설계할 때도 먼저 규칙 기반 자동화, 정책 엔진, 오케스트레이터만으로 처리 가능한 경로를 우선 설계합니다.
- AI는 기본 실행 레이어가 아니라 제한된 판단 레이어로 남아 있어야 합니다.

## Architecture principles
전체 구조는 다음 원칙을 따릅니다.

사용자/프론트 → 메인 AI → 오케스트레이터 → 툴/커넥터/실행기 → 결과 반환

필요한 경우 툴 내부에 보조 AI를 둘 수 있지만,
메인 흐름은 반드시 오케스트레이터가 통제합니다.

다음 원칙을 반드시 지킵니다.

- 기능 중심 앱이 아니라 확장 가능한 플랫폼으로 설계합니다.
- 툴이 직접 제각각 동작하지 않도록 합니다.
- 모든 실행은 중앙 오케스트레이터를 통해 관리합니다.
- 작업 상태, 실패, 재시도, 승인, 로그를 일관된 방식으로 다룹니다.
- 반복 가능한 작업은 규칙 기반 자동화로 우선 처리합니다.
- 새로운 기능은 플러그인/툴 형태로 추가 가능해야 합니다.

## Example capabilities
다음은 예시 기능이며, 구조는 범용적이어야 합니다.

- 크롤링 코드 자동 실행 후 결과 전달
- 구글 캘린더 조회 후 메시지 발송
- 업로드된 코드/파일 분석 후 필요한 런타임이나 의존성 판단
- 로컬 파일 읽기/쓰기 및 자동화
- 특정 프로그램 설치 또는 실행
- 향후 MCP, 외부 툴, 외부 AI 연동

## Security and approval rules
다음 보안 원칙을 반드시 고려합니다.

- 읽기 전용 작업은 자동 실행 가능
- 설치, 환경 변경, 외부 메시지 발송은 승인 후 실행
- 위험한 쉘 명령은 기본 차단
- 허용된 디렉토리, 명령, 설치 범위를 정책으로 관리
- 모든 실행은 로그와 결과를 남깁니다
- 시스템 변경 작업은 항상 검증 단계를 둡니다

## Expected output style
이 프로젝트에 대해 답변하거나 설계할 때는 항상 아래 순서를 우선합니다.

1. AI 없이 처리 가능한 부분 식별
2. 규칙 기반 자동화로 처리할 부분 분리
3. AI가 꼭 필요한 판단 지점 식별
4. 오케스트레이터 기준으로 흐름 설계
5. 툴/커넥터/실행기 책임 분리
6. 승인/정책/로그 구조 포함
7. MVP와 장기 확장 방향 함께 제시

## What to produce when asked
요청을 받으면 필요에 따라 아래를 설계하거나 생성합니다.

- 전체 아키텍처
- 폴더 구조
- 모듈 역할 분리
- 오케스트레이터 흐름
- 플러그인 인터페이스
- 정책 엔진 구조
- DB 스키마 초안
- API 스펙 초안
- MVP 범위
- 확장 로드맵
- 실제 코드 뼈대

## Constraints
다음 제약을 항상 기억합니다.

- "모든 것을 AI로 해결"하는 구조를 제안하지 않습니다.
- 반복 가능하고 안정적인 작업은 자동화 엔진 중심으로 설계합니다.
- AI는 판단 레이어이지, 기본 실행 레이어가 아닙니다.
- 비용, 속도, 안정성, 통제 가능성을 항상 함께 고려합니다.
