# 개발 워크플로우

이 프로젝트는 코드 품질 유지와 기획/구현/리뷰 단계를 분리하기 위해
3자 협업 개발 워크플로우를 사용합니다.

---

## 역할 분리

| 역할 | 에이전트 | 책임 |
|------|---------|------|
| 기획 | Claude Code (이 인스턴스) | 요구사항 분석, 아키텍처 결정, 구현 스펙 작성 |
| 구현 | Codex | `AGENTS.md` 기준 코드 + 테스트 작성 |
| 리뷰 | 별도 Claude Code 인스턴스 | 아키텍처 원칙 및 테스트 커버리지 기준 코드 리뷰 |

기획자는 프로덕션 코드를 직접 작성하지 않습니다.
구현자는 아키텍처 결정을 내리지 않습니다.
리뷰어는 수정을 직접 구현하지 않고 문제를 지적하여 Codex에 돌려보냅니다.

---

## 워크플로우 단계

```text
1. 기획 (Claude Code)
   - 기능 요청 또는 버그 리포트 수신
   - 현재 코드 분석 및 범위 파악
   - 구현 스펙 작성 (아래 템플릿 참고)
   - Codex에 스펙 전달

2. 구현 (Codex)
   - AGENTS.md와 스펙을 읽고 코드 + 테스트 구현
   - PR 오픈

3. 리뷰 (Claude Code)
   - 아래 리뷰 체크리스트 기준으로 PR 검토
   - 문제 발견 시 Codex에 피드백 반환
   - 모든 기준 통과 시 승인
```

## 구현 스펙 템플릿

Codex에 작업을 넘길 때 아래 형식을 사용합니다:

```markdown
## 스펙: <제목>

### 목표
한 줄 요약: 무엇을 왜 만드는지.

### 비AI 경로 (우선)
이 요청을 AI 없이 처리하는 규칙/플러그인/툴은?
- 규칙 패턴: `(정규표현식)`
- 툴/플러그인: `plugin_name`
- Params 형태: `{"key": "value"}`

### AI 폴백 경로
비AI 경로가 실패하거나 매칭되지 않을 때 AI는 어떻게 처리하는가?
- AI가 요청을 해석하는 방식
- AI가 폴백으로 호출할 툴

### 구현 위치
- 신규 파일: `backend/plugins/my_plugin/plugin.py`
- 수정 파일: `backend/intent/rules/rules.yaml` — `my_rule` 규칙 추가
- 수정 파일: 필요한 경우 `backend/plugins/loader.py`

### 테스트 요구사항
| 케이스 | 기대 결과 |
|--------|----------|
| 정확한 트리거 문구 매칭 | `my_plugin` 라우팅, `used_ai=False` |
| 부분 문구 매칭 | 동일 결과 |
| 무관한 명령 | 매칭 안 됨 |
| `run()` 유효한 params | `{"success": True, "data": {...}}` |
| `run()` params 누락 | 정상 처리 (오류 없이 graceful 처리) |

### 금지사항
- 건드리면 안 되는 것: `orchestrator/engine.py` (명시적 범위 외)
- 플러그인 내부에 AI 호출 금지
- 기존 테스트 파괴 금지
```

## 리뷰 체크리스트 {#review}

리뷰어 Claude Code로 PR을 검토할 때 아래 항목을 확인합니다:

### 아키텍처 준수

- [ ] AI 호출 전에 반드시 비AI 경로가 먼저 시도됨
- [ ] 툴 또는 플러그인 내부에서 AI를 호출하지 않음
- [ ] 오케스트레이터만이 순차적으로 스텝을 실행함
- [ ] 모듈 간 단방향 의존성 위반 없음: `gateway → intent → orchestrator → tools`

### 코드 품질

- [ ] `print()` 사용 없음 — logger 사용
- [ ] 하드코딩된 시크릿, 경로, 사용자 값 없음
- [ ] 모든 툴/플러그인 `run()`은 `{"success": bool, "data": ..., "error"?: ...}` 형태 반환
- [ ] 모든 비동기 함수는 `async def` / `await` 사용
- [ ] 새 함수 시그니처에 타입 힌트 있음

### 테스트 커버리지

- [ ] 새 플러그인: 라우팅 테스트 (올바른 툴 매칭), `run()` 테스트 (성공 + 실패 케이스)
- [ ] 새 툴: 유효한 params로 `run()` 테스트, 누락/잘못된 params 테스트
- [ ] 새 규칙: 패턴 매칭 테스트, 비매칭 테스트
- [ ] 새 오케스트레이터 로직: 스텝 실행 테스트, 에러 경로 테스트
- [ ] 기존 테스트 전부 통과 (회귀 없음)

### 스펙 준수

- [ ] 구현이 스펙에서 기술한 내용과 일치
- [ ] 스펙 범위를 넘는 기능 추가 없음
- [ ] 스펙의 금지사항이 준수됨

### Codex에 반환하는 경우

체크리스트 항목 하나라도 실패 시 반환합니다.
어떤 항목이 왜 실패했는지 구체적으로 명시합니다.
부분 구현은 승인하지 않습니다.

## 커뮤니케이션 패턴

### 기획 → Codex
위 템플릿 형식으로 스펙을 전달합니다.
정확한 파일 경로와 패턴/params 형태를 포함합니다.
건드리면 안 되는 것을 명시합니다.

### Codex → 리뷰어
PR 오픈 시:
- 제목: `feat: <내용>` / `fix: <내용>` / `refactor: <내용>`
- 본문: 스펙 링크, 변경된 파일 목록, 새 테스트 실행 방법

### 리뷰어 → Codex
수정 필요 시: 실패한 체크리스트 항목과 구체적인 라인 참조를 나열합니다.
승인 시: "리뷰 통과. 모든 체크리스트 항목 확인 완료."
