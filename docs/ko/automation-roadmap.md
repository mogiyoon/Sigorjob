# 자동화 로드맵

## 비전

**사용자가 원하는 모든 행동이 완전히 자동으로 실행되는** 시스템이 목표입니다.
일반적인 작업에서는 승인이나 추가 질문 없이 끝까지 실행됩니다.

참고 모델은 완전 자동화 에이전트 시스템(Open Interpreter / OpenClaw 스타일 실행)으로,
AI가 다단계 계획을 사용자 개입 없이 끝까지 실행하는 것입니다.
여기서의 핵심 차별점: AI를 더 강하게 만드는 게 아니라,
AI가 처리하던 경로를 더 빠르고 저렴한 비AI 자동화로 점점 교체하는 방식으로 도달합니다.

---

## 현재 상태 (2026년 4월)

```text
실행 경로별 커버리지:

규칙 기반 (비AI)          ████████░░  일반 요청의 ~65%
플러그인 기반 (비AI)       ████░░░░░░  플러그인 매칭 요청의 ~30%
AI 폴백                   ░░░░░░░░░░  대부분의 개방형 요청에 여전히 필요
완전 자동 (승인 없음)       ████░░░░░░  저위험 읽기 작업에서만 작동
```

현재 주요 한계:
- 대부분의 작업이 `medium` / `high` 위험 레벨에서 사람의 승인 필요
- 모호한 요청(단순한 것도)에 AI가 여전히 필요
- 스케줄러 제한 — 조건부 실행, 체이닝 미지원
- 외부 작업(캘린더 쓰기, 메시지 발송)은 커넥터 설정 필요

---

## Phase 1 — 비AI 규칙 커버리지 확장 (2026 Q2)

**목표**: AI 폴백 경로에 도달하는 요청 비율 감소.

### 구현할 것

| 작업 | 방법 |
|------|------|
| `rules.yaml`에 20+ 새 규칙 추가 | 자주 사용하는 모호한 문구를 기존 플러그인에 매핑 |
| 시간 표현 의도 정규화 추가 | "내일", "다음주 월요일", "오후 3시" → AI 없이 ISO 날짜시간 변환 |
| 연락처 참조 의도 정규화 | "민수에게", "팀장한테" → 수신자 이름 추출 |
| `shopping_helper` 규칙 확장 | 더 많은 구매/검색 문구 커버 |
| `calendar_helper` 규칙 확장 | 반복 일정, 다일 일정 커버 |
| `reminder_helper` 규칙 확장 | 자연어 시간 표현 커버 |
| `file_helper` 플러그인 추가 | 쉘 없이 자연어로 파일 읽기/쓰기/목록 |

**성공 지표**: AI 폴백 비율이 전체 라우팅 요청의 20% 미만으로 감소.

---

## Phase 2 — 컨텍스트 인식 자동 승인 (2026 Q3)

**목표**: 안전하고 예측 가능한 작업에 대한 수동 승인 제거.

### 구현할 것

| 작업 | 방법 |
|------|------|
| 승인 정책 엔진 | 툴별 규칙 정의: "알려진 캘린더에 calendar_helper 쓰기는 자동 승인" |
| 사용자 신뢰 이력 | 사용자가 이전에 승인한 툴 추적; 반복 패턴 자동 승인 |
| 위험 레벨 세분화 | 잘 이해된 플러그인 작업(캘린더 추가, 리마인더 설정) 위험 하향 조정 |
| 드라이런 모드 | 실행 내용을 미리 보여주고 N초 후 자동 실행 |
| 스케줄 작업 승인 바이패스 | 반복 작업은 최초 사람 승인 후 자동 승인 |

**성공 지표**: 일상 작업의 80% 이상이 승인 프롬프트 없이 실행됨.

---

## Phase 3 — 다단계 자율 실행 (2026 Q4)

**목표**: 시스템이 사용자를 방해하지 않고 다단계 계획을 끝까지 실행.

### 구현할 것

| 작업 | 방법 |
|------|------|
| 태스크 그래프 실행기 | 가능한 경우 병렬 실행; 조건부 분기 처리 |
| AI → 규칙 흡수 | AI가 특정 요청 유형을 N번 성공 처리 시 자동으로 규칙/플러그인 생성 |
| 커넥터 완성 | 연결된 계정으로 실제 캘린더 쓰기, 실제 메시지 발송, 실제 이메일 |
| 선제적 트리거 | 시간, 위치, 외부 이벤트 기반으로 시스템이 작업 시작 |
| 자가 복구 재시도 | 사용자에게 노출하기 전 조정된 params로 실패한 스텝 자동 재시도 |

**성공 지표**: "아침 루틴 설정해줘" 같은 다단계 목표를 기술하면 시스템이 중단 없이 모든 스텝 실행.

---

## Phase 4 — 플랫폼 자율성 (2027+)

**목표**: 시스템이 자체 자동화 포트폴리오를 관리.

### 구현할 것

| 작업 | 방법 |
|------|------|
| 자동화 학습 루프 | 사용자가 반복하는 것 관찰 → 새 규칙 제안 → 일회성 승인 |
| 플러그인 마켓플레이스 | 단일 승인으로 커뮤니티 플러그인 설치 |
| MCP 런타임 연동 | MCP 프로토콜로 외부 툴과 API 연결 |
| 크로스 디바이스 협업 | 모바일 트리거 → 로컬 PC 실행 → 결과를 모바일로 전달 |
| 장기 실행 백그라운드 에이전트 | 시간/일 단위로 실행되는 작업 (진행 상황 추적 포함) |

**성공 지표**: 시스템이 명시적 요청 없이도 사용자의 일상 자동화 필요의 50%+ 선제적 처리.

## 설계 원칙: AI 경로 흡수

AI가 요청을 성공적으로 처리할 때마다 묻습니다:
> "이 경로를 비AI 규칙이나 플러그인으로 교체할 수 있는가?"

흡수 사이클:

```text
1. AI가 새로운 요청 유형 처리
2. 시간에 따른 빈도 추적
3. 패턴이 명확해지면: 규칙 또는 플러그인 작성
4. 이후 동일 요청은 비AI 경로로 라우팅
5. 이 패턴에 대한 AI 호출 횟수 → 0
```

이것이 AI 비용을 낮게 유지하면서 커버리지를 확장하는 유일하게 지속 가능한 방법입니다.
목표는 "AI 제거"가 아니라 "AI가 증명한 것을 자동화로 흡수"입니다.

---

## 완전 자동화가 의미하지 않는 것

- 위험하거나 되돌릴 수 없는 작업에 대한 승인 제거가 아닙니다.
- AI가 모든 결정을 내리는 것이 아닙니다.
- 사용자가 시스템이 하는 일에 대한 가시성을 잃는 것이 아닙니다.

완전 자동화의 의미: **일상적이고, 예측 가능하며, 저위험인 작업이 마찰 없이 완료**됩니다.
위험하거나, 새롭거나, 중요한 작업은 여전히 사람의 검토를 거칩니다.
