# Runtime Surfaces

이 문서는 Sigorjob을 아래 세 환경에서 실행할 때의 차이를 설명합니다.

- 일반 브라우저
- 패키징된 Tauri 데스크톱 앱
- 모바일 앱

세 환경은 UI 코드를 많이 공유하지만, 완전히 같은 런타임은 아닙니다.

## 개요

### 브라우저

- 로컬 개발과 빠른 UI 확인에 가장 편함
- 일반 웹앱처럼 동작
- 링크, 클립보드, 새 탭 이동 같은 기본 브라우저 동작을 기대하기 쉬움

### Tauri 데스크톱 앱

- 최종 사용자용 패키징 데스크톱 환경
- release 빌드에서는 백엔드 sidecar를 자동 실행
- `127.0.0.1:8000` 고정이 아니라 실행 시 사용 가능한 로컬 포트를 동적으로 선택
- 겉보기엔 웹 UI지만, 브라우저 전제가 항상 통하지는 않음

### 모바일 앱

- 데스크톱 호스트에 연결되는 원격 제어 표면
- QR/수동 입력으로 연결하고 원격 UI를 WebView로 로드
- 가벼운 모바일 명령 입력 표면 역할도 가능
- 현재는 Android 쪽 공유 텍스트 흐름이 더 앞서 있음

## 핵심 차이

### 외부 링크 열기

브라우저:

- `target="_blank"`와 일반 앵커가 보통 기대대로 동작

Tauri:

- 외부 링크는 공용 helper나 Tauri 명령으로 여는 편이 안전
- 웹의 새 탭 전제를 그대로 믿으면 깨질 수 있음

모바일 WebView:

- 별도 의도가 없으면 WebView 내부 제약을 따라감

권장 방향:

- 외부 링크 동작을 UI 곳곳에 흩뿌리지 말고 공용 helper로 통일

### 클립보드

브라우저:

- `navigator.clipboard`가 자연스럽게 동작하는 경우가 많음

Tauri:

- 포커스, 권한, WebView 상태에 따라 다르게 보일 수 있음

모바일 WebView:

- 브라우저식 클립보드 가정이 더 불안정함

권장 방향:

- 클립보드도 raw 브라우저 API 대신 공용 helper로 감싸기

### 로컬 API 주소

브라우저 개발 환경:

- `http://127.0.0.1:8000`를 많이 전제함

패키징된 Tauri:

- 백엔드 포트가 동적임
- 프런트는 주입된 런타임 URL을 따라야 함

모바일:

- 로컬 루프백이 아니라 페어링된 데스크톱의 터널 URL을 사용

권장 방향:

- `127.0.0.1:8000`을 모든 환경의 진실처럼 취급하지 않기

### 인증

브라우저 로컬:

- 루프백 요청은 페어링 인증 없이도 가능

원격 브라우저/모바일:

- 페어링 토큰 bootstrap 후 cookie/bearer 인증 필요

권장 방향:

- 각 요청이 로컬 전용인지 원격 안전 경로인지 항상 분리해서 생각하기

## 현재 실무 가이드

UI 동작을 추가할 때는 이렇게 보면 좋습니다.

1. 이 동작이 일반 브라우저를 전제하고 있는가?
2. 패키징된 Tauri에서도 꼭 같아야 하는가?
3. 모바일 WebView는 같은 동작이 필요한가, 아니면 더 안전한 변형이 필요한가?

특히 주의할 예시:

- 외부 링크 열기
- 클립보드 복사
- 새 창 전제
- 브라우저 전용 API
- hostname 문자열 기반 런타임 감지

## 안전한 방향

- UI는 공유하되, 런타임 민감 동작은 helper로 분리
- 직접 브라우저 전역 객체를 만지기보다 capability-style helper를 선호
- 브라우저, Tauri, 모바일을 하나로 뭉뚱그리지 말고 실제로 다른 세 런타임으로 취급
